\subsection{Post-quantum Cryptography \& Motivation}

\begin{frame}
\frametitle{Concerns of Cryptography}
\topline
\titleline
There are five rudimentary concerns in information security:
\begin{itemize}
\item \textit{Confidentiality}: information must be kept private from unauthorized individuals.
\item \textit{Integrity}: information must not be altered by unauthorized individuals.
\item \textit{Availability}: information must be available for authorized individuals.
\item \textit{Authenticity}: information must have a verifiable source.
\item \textit{Non-repudiation}: the source of information must be publicly verifiable.
\end{itemize}
\end{frame}

%------------------------------------------------

\begin{frame}
\frametitle{Public-key Cryptography}
\topline
\titleline
The goal of cryptography is to define mathematically precise means of ensuring these information security goals. Proofs of cryptographic security depend, in many cases, on assumptions about the difficulty of solving some problem.\\
\vspace{20px}

Cryptographic protocols can be either \textit{private-key} or \textit{public-key} systems.\\
\vspace{20px}

Public-key systems require that every party takes ownership of both a public key ($pk$), the value of which is known by everyone on the network, and a private key ($sk$), known only to the owner of the keypair.
\vspace{20px}
\end{frame}

%------------------------------------------------

\begin{frame}
\frametitle{Quantum Cryptanalysis}
\topline
\titleline
A large-scale quantum computer will have the capability of breaking most modern public-key cryptosystems.\\
\vspace{25px}

This has lead to the development of the field known as post-quantum cryptography -- the aim of which is to develop cryptosystems resistant to quantum cryptanalysis.
\end{frame}

%------------------------------------------------

\begin{frame}
\frametitle{Performance of Post-quantum Cryptosystems}
\topline
\titleline
Common approaches to post-quantum cryptography include
\begin{itemize}
  \setlength\itemsep{1em}
  \item Lattice-based cryptography,
  \item Hash-based cryptography,
  \item Multivariate-based cryptography,
  \item Code-based cryptography, and
  \item Isogeny-based cryptography.
\end{itemize}
\end{frame}

\begin{frame}
\frametitle{Torsion Subgroups}
\topline
\titleline

We write $E[r]$ to denote the set of all points on a curve $E$ with order $r$, e.g;
$$
E[r] = \{P \in E: [m]P = \mathcal{O} \}
$$
\end{frame}

\begin{frame}
\frametitle{Isogenies}
\topline
\titleline
Isogenies are maps that take a point on one elliptic curve to a point on another.
For an isogeny $\phi$ mapping from $E_1$ to $E_2$, we can write
$$
\phi: E_1 \rightarrow E_2
$$
These maps have the following two properties
\begin{itemize}
  \item $\phi(\mathcal{O}) = \mathcal{O}$
  \item $\phi(P^{-1}) = (\phi(P)^{-1})$
\end{itemize}
\end{frame}

%------------------------------------------------

\begin{frame}
\frametitle{Isogenies}
\topline
\titleline

\begin{tcolorbox}
\begin{lemma}[Uniquely identifying isogenies]
\label{lem:isogkern}
Let $E$ be an elliptic curve and let $\Phi$ be a finite subgroup of $E$. There is a unique elliptic curve $E'$ and a seperable isogeny $\phi: E \rightarrow E'$ satisfying $\text{ker}(\phi) = \Phi$.
\end{lemma}
\end{tcolorbox}

\end{frame}

\subsection{Isogeny-based Signatures}

\begin{frame}
\frametitle{Interactive Identification Schemes}
\topline
\titleline

Identification schemes are used to confirm the identity of a user on a network. These protocols are typically composed by the tuple of algorithms (\keygen, \commit, \prove, \verify).\\

For \bob to prove his identity to \alice, a protocol of this type would run as follows:
\begin{enumerate}
\item \bob runs \keygen($1^\lambda)$ to generate his keypair $(sk,pk)$.
\item \bob runs \commit() to generate $com$ and sends it to \alice.
\item \alice sends a randomly generated ``challenge" value $ch \in \omega$ and sends it to \bob.
\item \bob runs \prove($sk$, $com$, $ch$) with output $resp$, the response to \alice's challenge.
\item \alice runs \verify($pk$, $com$, $ch$, $resp$) with output $b \in {0,1}$. \bob has successfully proven his identity to \alice if $b = 1$.
\end{enumerate}

\end{frame}

\begin{frame}
\frametitle{Isogeny-based Proof of Identity}
\topline
\titleline
The isogeny-based proof of identity protocol outlined by De Feo et al. consists of the algorithms \keygen, \commit, \prove, and \verify.\\

\vspace{25px}
In the scheme, \bob can prove to \alice (or vice-versa) that he is the owner of the keypair $(pk, sk)$ by forming many SIDH secret agreements with an arbitrary, random entity \randall, and showing that he can always come to a valid shared secret with \randall.
\end{frame}

%------------------------------------------------

\begin{frame}
\frametitle{Isogeny-based Proof of Identity}
\topline
\titleline
\commit: \bob generates \randall's random point $\cyr \in E[\laea]$ and the corresponding isogeny $\pr$. \bob then performs \secagr with \randall and sets $com = (E/\langle \cyr \rangle, E/\langle \rb, \cyr \rangle)$.\\

\vspace{15px}
\textit{Challenge:} \alice responds with a random challenge bit $ch \in \{0,1\}$.\\

\vspace{15px}
\prove: If $ch = 0$ then \bob reveals isogenies $\pr$ and $\prp: E/\langle \rb \rangle \rightarrow E/\langle \rb,\cyr \rangle$. If $ch = 1$ then \bob reveals $\pbp: E/\langle \cyr \rangle \rightarrow E/\langle \rb,\cyr \rangle$.\\

\vspace{15px}
\verify: If $ch = 0$ return 1 iff $\cyr$ and $\pb(\cyr)$ have order $\laea$ and generate the kernels of isogenies from $E \rightarrow E/\langle \cyr \rangle$ and $E \rightarrow E/\rangle \rb,\cyr \rangle$ respectively. If $ch = 0$ return 1 iff $\pr(\rb)$ has order $\lbeb$ and generates the kernel of an isogeny over $E/\langle \cyr \rangle \rightarrow E/\langle \rb,\cyr \rangle $.\\

\end{frame}

\begin{frame}
\frametitle{Signature Schemes}
\topline
\titleline
Signature schemes are used to prove that a particular party supplied and authenticated a given message $m$, and that $m$ had not been edited by an unauthorized party. These schemes consist of the algorithms \keygen, \sign, and \verify.\\

\vspace{15px}
If \bob wishes to authenticate a message $m$ and send it to \alice, the following will take place:
\begin{enumerate}
  \item \bob runs \keygen to produce his keypair $(pk_{\rb}, sk_{\rb})$.
  \item \bob runs \sign($m$, $sk_{\rb}$) to produce a signature $\sigma$, and sends ($m$, $\sigma$) to \alice.
  \item \alice runs \verify($m$, $\sigma$, $pk_{\rb}$) returning 1 if \bob's message was successfully verified and 0 otherwise.
\end{enumerate}

\end{frame}

\begin{frame}
\frametitle{Fiat-Shamir Transform}
\topline
\titleline
The Fiat-Shamir transform is a process by which an interactive identification scheme can be turned into a signature scheme.\\

\vspace{20px}
The transform works by first circumventing the need for challenge input from the verifier, turning it into a non-interactive identification scheme.\\

\vspace{20px}
Then, the message $m$ is factored into the identification protocol, allowing the prover to verify not just their own identity, but the authenticity of a message as well.\\
\end{frame}

\begin{frame}
\frametitle{Applying Fiat-Shamir To an IPoI}
\topline
\titleline

Returning to our outline for a generic, interactive identification protocol, we would make the following changes:\\

\vspace{10px}
\begin{enumerate}
  \item \bob runs \keygen($1^\lambda)$ to generate his keypair $(sk,pk)$.
  \item \bob runs \commit() to generate $com$ and sends it to \alice.
  \item \sout{\alice sends a randomly generated ``challenge" value $ch \in \omega$ and sends it to \bob.}
  \setcounter{enumi}{2}
  \item \bob computes the challenge value as $ch = H(com, m)$ for some cryptographically secure hash function $H$ and for message $m$.
  \item \bob runs \prove($sk$, $com$, $ch$) with output $resp$, the response to \alice's challenge.
  \item \alice runs \verify($pk$, $com$, $ch$, $resp$) with output $b \in {0,1}$. \bob has successfully proven his identity to \alice if $b = 1$.
\end{enumerate}
\end{frame}

\subsection{SIDH Public Key Compression}

\begin{frame}
\frametitle{SIDH Public Key Compression}
\topline
\titleline
Azerderakhsh et al. showed that SIDH public keys can be compressed in the following way.\\

\vspace{10px}
Take \alice's public key $pk = (E_{\ba}, \pa(\genpb), \pa(\genqb))$.

By generating a basis $\{R_1, R_2\}$ for $E_{\ba}[\lbeb]$ we can represent the point components of $pk$ as
$$
\genpa = \alpha_P R_1 + \beta_P R_2
$$
$$
\genqa = \alpha_Q R_1 + \beta_Q R_2
$$

Giving $pk = (E_{\ba}, \alpha_P, \beta_P, \alpha_Q, \beta_Q)$. Public keys in this form are $4\log p$ bits compared to the usual $6'log p$ bits.
\end{frame}

%------------------------------------------------

\begin{frame}
\frametitle{SIDH Public Key Compression}
\topline
\titleline
Costello et al. then showed that $(E_{\ba}, \alpha_P, \beta_P, \alpha_Q, \beta_Q)$ could be further compressed to $(E_{\ba}, b, \zeta, \alpha, \beta)$, where $b \in \{1,0\}$.


\end{frame}
